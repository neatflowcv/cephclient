# ADR 0008: cli 패키지 테스트는 작성하지 않는다

- Status: Accepted
- Date: 2026-04-03

## Context

이 프로젝트의 `internal/app/cli` 패키지는 `kong` 구조체 선언과 얇은 실행 위임으로
구성된다.

CLI 계층에서 직접 소유한 로직은 많지 않고, 실제 확인이 필요한 부분은 개발자가
명령을 직접 실행해도 빠르게 검증할 수 있다.

CLI 테스트를 추가하면 다음 비용이 생긴다.

- parser 조립과 에러 메시지 같은 라이브러리 세부 동작에 테스트가 결합된다.
- 테스트를 위해 공개 API나 seam을 추가하고 싶어지는 압력이 생긴다.
- flow, podman 등 하위 계층에서 이미 보장하는 동작을 다시 반복 검증하게 된다.

## Decision

이 프로젝트에서는 `internal/app/cli` 패키지에 자동화 테스트를 작성하지 않는다.

CLI 변경 검증은 기본적으로 실제 명령을 직접 실행하는 수동 확인으로 수행한다.

CLI 패키지를 테스트하기 위한 공개 API, 테스트 전용 seam, parser 노출 함수도 추가하지 않는다.

## Rationale

- CLI 계층은 얇아서 자동화 테스트의 이득보다 유지비가 더 크다.
- 실제 사용 방식 그대로 명령을 실행하는 편이 확인 경로가 더 단순하다.
- 테스트 편의를 위해 프로덕션 API를 넓히지 않으면 구조가 더 깔끔하게 유지된다.
- 자동화 테스트는 비즈니스 로직과 외부 호출을 담당하는 하위 레이어에 집중하는 편이 효율적이다.

## Consequences

### Positive

- CLI 패키지 코드가 테스트 보조 구조 없이 단순하게 유지된다.
- parser 라이브러리 세부 구현 변화에 덜 민감해진다.
- 테스트는 flow, podman, domain 같은 실제 로직 계층에 집중된다.

### Negative

- CLI 인자 구조 변경은 자동화된 회귀 테스트로 바로 잡히지 않는다.
- 수동 실행 확인을 빼먹으면 명령 구조 회귀를 늦게 발견할 수 있다.

## Alternatives Considered

### CLI parser 테스트를 최소한으로 유지

명령 구조 회귀를 자동으로 잡을 수는 있지만, 이 프로젝트에서는 그 이득보다
테스트 유지와 API 노출 비용이 더 크다고 판단한다.

### 테스트 전용 seam을 두고 `RunWithArgs`를 검증

조립 경로를 자동화할 수 있지만, 테스트 때문에 프로덕션 구조를 바꾸게 되어
기본 원칙에 맞지 않는다.
