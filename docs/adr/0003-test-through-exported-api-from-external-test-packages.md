# ADR 0003: 외부 테스트 패키지에서 exported API 중심으로 검증

- Status: Accepted
- Date: 2026-03-30

## Context

이 프로젝트는 CLI, 애플리케이션 흐름, 인프라 구현을 계층으로 나누어 유지하려고 한다.

테스트를 작성할 때 패키지 내부 구현 세부사항에 직접 접근하면 테스트는 빨라질 수 있지만, 리팩터링 저항이 커지고 패키지 경계가 흐려질 수 있다.

테스트 정책을 정할 때 다음 조건이 중요하다.

- 패키지 외부에서 보이는 계약을 기준으로 동작을 검증해야 한다.
- 내부 헬퍼나 비공개 구조에 대한 결합을 줄여야 한다.
- 리팩터링 시 테스트가 구현이 아니라 계약을 보호해야 한다.

## Decision

기본 테스트 정책은 외부 테스트 패키지(`package foo_test`)를 사용하고,
exported API만 통해 검증하는 것으로 한다.

테스트를 위해 필요한 주입 지점이 있으면 비공개 구현에 직접 접근하는 대신
exported 생성자, 인터페이스, 또는 테스트 가능한 주입 포인트를 추가한다.

## Rationale

- 테스트가 패키지 사용자 관점의 계약을 검증하게 된다.
- 비공개 함수와 내부 필드 변경이 테스트 대규모 수정으로 이어지는 일을 줄일 수 있다.
- 계층 경계를 코드뿐 아니라 테스트에서도 일관되게 유지할 수 있다.
- 공개 API 설계가 테스트 가능성을 자연스럽게 반영하도록 유도한다.

## Consequences

### Positive

- 리팩터링 내성이 좋아지고 구현 세부사항 노출이 줄어든다.
- 패키지 경계와 공개 계약이 더 명확해진다.
- 테스트가 실제 사용 경로와 가까워진다.

### Negative

- 비공개 함수 단위의 매우 세밀한 테스트는 직접 작성하기 어렵다.
- 테스트를 위해 주입 가능한 exported API를 추가해야 할 수 있다.
- 일부 오류 경로나 예외 상황 검증을 위해 테스트용 훅 설계가 필요할 수 있다.

## Alternatives Considered

### 내부 테스트 패키지 사용

비공개 함수와 구조에 직접 접근할 수 있어 세밀한 테스트가 쉬워진다. 하지만
구현 결합도가 높아지고 공개 계약보다 내부 구조를 보호하는 테스트가 되기
쉽다.

### 기능별로 정책 혼용

상황에 따라 유연하게 쓸 수는 있지만, 팀 차원의 일관된 기준을 세우기
어렵다. 이 프로젝트는 우선 외부 테스트 패키지와 exported API 검증을
기본값으로 둔다.
