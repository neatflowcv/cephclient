# ADR 0004: 테스트는 AAA 구조와 testify/require를 기본으로 사용

- Status: Accepted
- Date: 2026-03-30

## Context

이 프로젝트는 테스트를 빠르게 읽고, 실패 원인을 바로 파악할 수 있는
일관된 작성 규칙이 필요하다.

테스트 코드 스타일을 정할 때 다음 조건이 중요하다.

- 테스트 본문 구조가 눈에 잘 들어와야 한다.
- 실패 시 어떤 전제와 어떤 검증이 깨졌는지 분명해야 한다.
- 표준 라이브러리만으로 반복되는 오류 처리와 단정 보일러플레이트를
  줄일 수 있으면 좋다.

패턴이 섞이면 테스트마다 읽는 방식이 달라지고, 작은 단정도 장황해질 수
있다.

## Decision

테스트는 기본적으로 AAA(Arrange, Act, Assert) 구조를 유지한다.

단정과 실패 처리는 `github.com/stretchr/testify/require`를 기본으로 사용한다.

테스트 본문은 준비, 실행, 검증의 흐름이 드러나도록 작성하고, 검증 단계에서
즉시 중단이 적절한 경우 `require`를 사용한다.

테스트에서 context가 필요하면 `context.Background()` 대신 `t.Context()`를
사용한다.

## Rationale

- AAA 구조는 테스트의 의도를 빠르게 파악하게 해 준다.
- Arrange, Act, Assert 경계를 유지하면 테스트가 커져도 읽기 쉽다.
- `require`는 오류 확인과 값 비교를 간결하게 만들어 중복을 줄인다.
- 실패 시 첫 전제 위반 지점에서 바로 멈춰 후속 오류 노이즈를 줄일 수 있다.
- 테스트 생명주기와 함께 취소되는 `t.Context()`를 쓰면 테스트 컨텍스트 관리가
  더 명확해진다.

## Consequences

### Positive

- 테스트 스타일이 통일되어 리뷰와 유지보수가 쉬워진다.
- 실패 메시지와 단정 코드가 더 간결해진다.
- 테스트가 절차적으로 읽혀 의도 파악이 빨라진다.
- 테스트에서 사용하는 context의 수명과 취소 시점이 `testing.T`와 정렬된다.

### Negative

- 팀이 `testify/require` 의존성을 받아들여야 한다.
- 단순 비교도 `require` 사용 방식에 맞춰 작성해야 한다.
- 모든 테스트에 엄격한 AAA 구분을 적용하면 아주 짧은 테스트는 다소 형식적으로
  보일 수 있다.
- 오래된 Go 버전에서는 `t.Context()`를 사용할 수 없으므로 최소 Go 버전 가정이
  필요하다.

## Alternatives Considered

### 표준 라이브러리만 사용

의존성을 줄일 수 있지만, 반복적인 오류 처리와 비교 코드가 늘어나기 쉽다.

### assertion 스타일 자유 선택

상황별 유연성은 있지만, 테스트마다 구조와 단정 스타일이 달라져 일관성이
떨어질 수 있다.
