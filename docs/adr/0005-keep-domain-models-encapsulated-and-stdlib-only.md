# ADR 0005: domain 모델은 캡슐화하고 표준 라이브러리에만 의존

- Status: Accepted
- Date: 2026-03-30

## Context

이 프로젝트는 CLI, 애플리케이션 흐름, 인프라 구현을 분리하고, 도메인 모델을
그 중심에 두려고 한다.

도메인 패키지를 설계할 때 다음 조건이 중요하다.

- 도메인 모델은 다른 계층의 구현 세부사항에 오염되지 않아야 한다.
- 외부 라이브러리 변경이 도메인 타입 정의에 직접 영향을 주지 않아야 한다.
- 도메인 상태 변경과 조회는 명시적인 계약을 통해 이뤄져야 한다.

필드를 외부에 직접 노출하면 호출자가 내부 표현에 결합되고, 나중에 검증이나
불변식이 필요해질 때 변경 비용이 커질 수 있다.

## Decision

`internal/pkg/domain` 패키지는 Go standard library 외의 의존성을 갖지 않는다.

domain 구조체의 필드는 외부에서 직접 접근할 수 없도록 비공개로 둔다.

domain 객체는 생성자 함수와 exported 메서드를 통해 생성하고 조회한다.

즉, 호출자는 `BucketStats.ID` 같은 필드 직접 접근 대신 `NewBucketStats(...)`,
`ID()` 같은 명시적인 API를 사용한다.

## Rationale

- 도메인 모델을 외부 계층의 표현 방식에서 분리할 수 있다.
- 내부 필드 표현을 바꿔도 공개 메서드 계약만 유지하면 된다.
- 생성자에서 유효성 검사나 불변식 적용 지점을 확보할 수 있다.
- 도메인 패키지가 가장 안정적인 계층으로 남아 재사용성과 테스트 용이성이
  높아진다.

## Consequences

### Positive

- 도메인 모델의 캡슐화가 강화된다.
- 외부 계층은 필드 구조가 아니라 도메인 계약에 의존하게 된다.
- 외부 라이브러리 변경이 domain 패키지로 침투하는 것을 막기 쉽다.

### Negative

- 단순 데이터 구조도 생성자와 getter 메서드가 필요해 코드가 늘어날 수 있다.
- 빠른 프로토타이핑 단계에서는 다소 번거롭게 느껴질 수 있다.
- 직렬화나 매핑 시 명시적인 변환 코드가 더 필요할 수 있다.

## Alternatives Considered

### 공개 필드 구조체 사용

작성과 접근은 단순하지만, 호출자가 내부 표현에 직접 결합되고 캡슐화가 약해진다.

### domain 패키지에 외부 라이브러리 타입 허용

초기 구현은 빨라질 수 있지만, 도메인 모델이 인프라 또는 프레임워크 세부사항에
오염될 수 있다. 이 프로젝트는 domain 계층을 가장 안정적인 계층으로 유지한다.
