# ADR 0007: mock framework는 moq를 사용한다

- Status: Accepted
- Date: 2026-03-30

## Context

이 프로젝트는 인터페이스를 협력자 경계로 두고 테스트를 작성한다.

mock framework를 선택할 때 다음 조건이 중요하다.

- 테스트 코드에서 함수 시그니처와 인자 타입이 분명하게 드러나야 한다.
- 메서드 이름 문자열이나 느슨한 런타임 설정보다 컴파일 타임 타입 체크를
  더 신뢰할 수 있어야 한다.
- 생성된 mock이 단순해서 테스트 의도를 읽기 쉬워야 한다.

`testify/mock` 같은 동적 방식은 빠르게 작성할 수 있지만, 메서드 이름과 반환값
설정 일부가 런타임에 더 많이 검증된다.

`gomock`은 expectation 선언이 강력하지만, 현재 프로젝트가 원하는 기준은
엄격한 expectation DSL 자체보다 타입이 분명한 함수 필드 기반 mock에 더 가깝다.

## Decision

이 프로젝트의 mock framework 기본 선택은 `github.com/matryer/moq`로 한다.

인터페이스 mock은 기본적으로 `moq`로 생성하고, 테스트에서는 생성된 mock의
함수 필드와 호출 기록 API를 사용한다.

## Rationale

- `moq`는 인터페이스 메서드 시그니처를 함수 필드로 그대로 노출해서 타입 체크가
  분명하다.
- 인자와 반환값 타입이 맞지 않으면 컴파일 단계에서 바로 드러난다.
- 메서드 이름 문자열 기반 설정보다 리팩터링에 더 안전하다.
- 생성된 mock 구조가 단순해서 테스트에서 어떤 동작을 주입하는지 바로 읽힌다.
- 호출 횟수는 `Calls()` 기록으로 확인할 수 있어 필요한 검증을 충분히 표현할 수 있다.

## Consequences

### Positive

- mock 설정 코드에서 타입 정보가 더 직접적으로 드러난다.
- 인터페이스 변경 시 깨지는 지점이 컴파일 단계에서 더 빨리 보인다.
- 생성된 mock이 단순해서 디버깅과 리뷰가 쉽다.

### Negative

- `gomock`처럼 “정확히 1회 호출”을 expectation DSL로 선언하는 표현력은 약하다.
- 호출 순서나 복잡한 matcher가 중요한 테스트에서는 추가 검증 코드가 필요할 수 있다.
- mock 생성 파일을 관리하는 규칙이 필요하다.

## Alternatives Considered

### `testify/mock` 사용

기존 `testify/require`와 함께 쓰기 쉽지만, 메서드 이름 문자열과 런타임 설정 비중이
더 크다.

### `gomock` 사용

호출 횟수와 순서 검증은 더 강력하지만, 이 프로젝트에서는 expectation DSL보다
타입이 분명한 함수 필드 기반 mock이 더 우선이다.
