# ADR 0006: 중복되는 CLI run 경로 테스트를 위해 테스트 전용 seam을 추가하지 않는다

- Status: Accepted
- Date: 2026-03-30

## Context

이 프로젝트의 CLI는 `internal/app/cli`에서 명령 구조체를 정의하고, `RunWithArgs`
에서 `kong` 파서를 생성해 실행한다.

`list-buckets`를 추가하면서 `RunWithArgs` 경로 전체를 테스트하기 위해 서비스 주입용
seam을 두고 parser 실행 테스트를 작성할 수 있었다. 하지만 이 방식은 다음 문제를
만든다.

- 명령 객체 자체의 작은 로직보다 `RunWithArgs` 조립 코드까지 함께 테스트하게 된다.
- 이미 flow, podman 테스트에서 검증하는 동작을 다시 CLI run 경로에서 반복하게 된다.
- 테스트를 위해 공개 API나 seam을 추가하면 실제 프로덕션 구조보다 테스트 편의가
  우선된 설계가 된다.

현재 CLI 계층에서 새로 추가되는 로직은 대체로 얇은 위임과 출력 포맷이며, parser
라이브러리 자체 동작을 프로젝트가 반복 검증할 필요는 크지 않다.

## Decision

중복되는 parser/run 경로 테스트를 위해 `RunWithArgs` 주변에 테스트 전용 seam을
추가하지 않는다.

`internal/app/cli/run_test.go` 같은 조립 경로 테스트는 기본적으로 두지 않는다.

CLI 테스트가 필요할 때는 다음 원칙을 따른다.

- 명령 자체에 의미 있는 출력 규칙이나 분기 로직이 있을 때만 그 로직을 직접 검증한다.
- 단순 위임과 parser 조립은 `kong` 사용과 기존 구조를 신뢰하고 별도 테스트로 반복하지 않는다.
- 실제 행위 검증은 flow, podman 등 하위 레이어 테스트에서 담당한다.

## Rationale

- 테스트 대상은 프로젝트가 직접 소유한 로직에 집중하는 편이 유지보수 비용이 낮다.
- 테스트 전용 seam을 줄이면 공개 API와 내부 구조가 단순해진다.
- 동일한 성공/실패 경로를 여러 레이어에서 반복 검증하지 않아도 핵심 동작은 충분히
  보장된다.

## Consequences

### Positive

- CLI 조립 코드가 더 단순해진다.
- 테스트가 레이어별 책임에 더 가깝게 정리된다.
- parser 교체나 내부 배선 변경이 있어도 불필요한 테스트 수정이 줄어든다.

### Negative

- `RunWithArgs` 전체 경로를 직접 검증하는 테스트는 기본적으로 없어진다.
- 아주 얇은 CLI 출력 로직은 테스트보다 코드 리뷰와 하위 레이어 검증에 더 의존하게 된다.

## Alternatives Considered

### `RunWithArgs`에 서비스 주입 seam을 유지하고 run 경로를 테스트

구현은 가능하지만, 프로덕션 코드에 테스트 편의용 API를 남기고 중복 검증을 늘린다.

### 모든 CLI 명령에 parser 수준 테스트를 추가

회귀를 더 넓게 잡을 수는 있지만, 현재 프로젝트 규모와 CLI 복잡도 대비 비용이 크다.
