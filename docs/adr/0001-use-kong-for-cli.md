# ADR 0001: CLI 라이브러리로 kong 채택

- Status: Accepted
- Date: 2026-03-30

## Context

이 프로젝트는 CLI 진입점과 옵션 파싱 구조를 정해야 한다.

CLI 라이브러리를 선택할 때 다음 조건이 중요하다.

- 선언적인 방식으로 명령과 옵션을 정의할 수 있어야 한다.
- 애플리케이션 상태를 전역 변수에 의존하지 않고 구성할 수 있어야 한다.
- 테스트 코드에서 명령 파싱과 실행을 분리해 다루기 쉬워야 한다.

일부 CLI 라이브러리는 패키지 전역 상태나 암묵적인 전역 커맨드 객체에
기대는 사용 패턴이 흔하다. 이런 방식은 초기 구현은 빠를 수 있지만, 테스트
격리와 의존성 주입 측면에서 제약이 생기기 쉽다.

## Decision

CLI 라이브러리로 `kong`을 채택한다.

명령, 옵션, 실행 대상을 구조체로 정의하고, 필요한 의존성은 명시적으로 주입하는 방식을 기본 원칙으로 삼는다.

## Rationale

- `kong`은 구조체 기반 선언이 자연스러워 CLI 스키마를 코드로 읽기 쉽다.
- 패키지 전역 변수에 기대지 않아도 자연스럽게 사용할 수 있다.
- 파서 생성과 실행 흐름을 명시적으로 제어할 수 있어 테스트 작성이 수월하다.
- 명령별로 필요한 의존성을 분리하기 쉬워 확장 시 결합도를 낮출 수 있다.

## Consequences

### Positive

- 글로벌 상태 없이 CLI 구성을 유지할 수 있어 코드 구조가 단순해진다.
- 테스트에서 독립적인 parser/context를 만들기 쉬워진다.
- 하위 명령이 늘어나도 구조체 중심으로 확장하기 쉽다.

### Negative

- 팀이 `kong`의 태그 기반 선언 방식에 익숙해져야 한다.
- 다른 대중적인 CLI 라이브러리보다 참고 예제가 상대적으로 적을 수 있다.

## Alternatives Considered

### `flag`

표준 라이브러리라는 장점이 있지만, 하위 명령과 복잡한 옵션 구조를 다루는 코드는 금방 장황해질 수 있다.

### 전역 커맨드 객체 기반 라이브러리

빠르게 시작하기는 쉽지만, 전역 상태에 기대는 패턴이 섞이기 쉽다. 이 프로젝트에서는 글로벌 변수를 사용하지 않는 구성을 우선한다.
